# Harness Engineering Checklist

本清单用于 backend 日常需求开发、缺陷修复和接口改造，目标是让改动具备可验证、可观测、可回滚的工程闭环。

## 1. 需求与边界

- 明确输入、输出、调用方、依赖服务、配置来源。
- 明确正常路径、失败路径、重试路径、幂等要求。
- 明确状态源和状态流转，避免只修表象入口。
- 明确兼容性影响：接口、数据结构、配置、数据库、调用方。

## 2. 实现原则

- 优先修根因，不做只覆盖单点现象的补丁。
- 优先保持现有分层：API 只编排，业务逻辑落在 Application，持久化落在 Infrastructure。
- 优先最小闭环实现：功能、状态、日志、验证一起交付。
- 不混入无关重构，不回退用户已有改动。

## 3. 可靠性护栏

- 明确定义终态与中间态，并为终态补状态回写。
- 异步回调、轮询确认、重复请求要保证幂等。
- 外部依赖失败时要有可解释的错误和日志。
- 对非法参数、鉴权失败、资源不存在、冲突状态返回正确语义，而不是统一 500。

## 4. 可观测性

- 关键链路记录结构化日志：用户、订单、资源标识、状态、失败原因。
- 日志能回答“谁调用了什么、结果如何、失败在哪一步”。
- 必要时补充健康检查、状态查询或最小诊断入口。

## 5. 验证与发布安全

- 每次改动至少做一项最小可行验证：build、test、type check 或链路验证。
- 明确已验证项、未验证项和剩余风险。
- 配置和行为变更需要同步更新文档或说明。
- 改动应具备回滚路径，避免把高风险 schema、接口和行为变更一次性耦合。

## 6. 交付输出

- 先给结论，再给关键改动、验证结果、剩余风险。
- 不把“理论上可行”表述成“已验证通过”。
- 文件定位、接口影响、验证命令都应能被快速复查。
